What is agentic AI?

Most of what you’ve probably experienced with LLMs is chat-based: you write a prompt, the system responds, the conversation ends. You are fundamentally in control. You direct each exchange.

Agentic systems work differently. You describe a goal or task in natural language—“find all mentions of magistrates in this database, translate them, produce a summary table”—and the agent plans and executes a series of steps to achieve it. It reads files, writes files, runs code, searches the web, evaluates the results, and decides what to do next, all without waiting for you to intervene between each step. The agent has a loop: it takes an action, observes the outcome, and determines the next action based on what happened.

This is a qualitative shift, not merely a quantitative one. A chat response is atomic: you ask, you receive, you evaluate. An agentic workflow is procedural: the system maintains context across multiple steps, modifies your files, accumulates results, and iterates toward completion. The agent isn’t just answering your question; it’s trying to solve a problem.

The capabilities are real. A chat LLM can translate a Latin inscription if you give it the text. An agentic system can search a directory of photographs for inscriptions matching criteria, extract text from images, translate them, cross-reference with a database, and compile results—all in one workflow. The ability to loop, use tools, and modify files fundamentally expands what’s possible.

But this expansion comes with new challenges. When you review a chat response, you can quickly check whether it’s correct. When an agent has modified twenty files, you can’t as easily review every change. The autonomy that makes the tool powerful also makes it harder to maintain the oversight that makes the output trustworthy. There is a tension between speed and verification that doesn’t have a clean resolution.

Agentic tools in practice

Several agentic systems are currently available or emerging. Here are the most relevant for humanities scholars (as of February 2026):

Claude Code (Anthropic)

Claude Code is Anthropic’s command-line tool that gives Claude direct access to your file system, terminal, and development environment. You describe what you want in natural language; it reads your files, writes code, runs tests, iterates, and modifies your workspace. Designed for software development, but increasingly used for data processing, document generation, and research workflows. You maintain full control of your files (they stay on your machine unless you explicitly share them), and you can undo changes via version control. Requires an Anthropic API key or subscription.

Claude Code on the web (Cowork)

The same agentic capabilities are available through a web interface without needing to install command-line tools or configure local environments. This is suitable for users who want to experiment with agentic workflows from a browser without the overhead of local setup. Still requires an API key or subscription.

Other agentic tools (brief landscape)

The agentic AI landscape is evolving rapidly. Other tools worth noting:

  • GitHub Copilot (Microsoft/GitHub) — AI coding assistant integrated into development environments, focused primarily on code completion and generation.
  • Cursor — An AI-native code editor combining agentic capabilities with a native development environment.
  • Devin (Cognition) — Autonomous software engineering agent designed to handle end-to-end development tasks.

This field is moving quickly. Tools and capabilities available in early 2026 will likely differ from what’s current by the time you read this. What matters less than any specific tool is understanding the underlying pattern: agents that combine language models with tool access and iteration loops, grounded in your own files and data rather than general knowledge.

Getting started with agentic tools

If you’re not a programmer, don’t think of agentic tools as programming aids. Think of them as workers who can manipulate files, run code, and search the web under your direction. You don’t need to understand what they’re doing under the hood any more than you need to understand how a spreadsheet calculates a formula. You need to understand what they can do, how to direct them, and how to verify the output.

What you need

For Claude Code, you need an API key or subscription through Anthropic, and terminal access to your computer. For Cowork (web-based), you need a browser and a subscription. Either way, you need to be comfortable giving the tool access to your file system.

The basic interaction model

You describe a task in natural language. The agent reads relevant files, plans an approach, executes steps, checks results, and iterates. You watch the process unfold and can interrupt if something looks wrong. Once complete, you review the output and either accept it, ask for revisions, or start over.

The process is conversational, but with a difference: the agent is actually doing things, not just talking about them.

Key principles for working with agents

1. Start small. Give the agent a contained, well-defined task before attempting something complex. “Convert this bibliography from Chicago style to Harvard” before “restructure my entire project directory and update all references.” Small tasks let you calibrate how the agent works, understand its failure modes, and build confidence before investing in larger efforts.

2. Use version control. Agents modify files. If the agent makes mistakes—and it will—you need a way to review what changed and undo it. Git (or another version control system) is essential. Commit before giving the agent significant tasks, so you have clean checkpoints. Review diffs after the agent finishes to understand what changed.

3. Review everything. Don’t assume the agent got it right just because the process completed without errors. An agent can produce syntactically valid code, run successfully, and still produce wrong results. Check the output, spot-check results against source data, and verify that the agent didn’t make unintended changes elsewhere. If you’re trusting the tool with important work, spend time on verification.

4. Be specific about constraints. The clearer your instructions, the better the results. “Use only the sources in this folder,” “don’t modify files outside this directory,” “check with me before deleting anything.” Explicit constraints reduce unpleasant surprises.

5. Treat the agent as a collaborator, not an oracle. The best results come from iterative dialogue. You describe what you want, the agent produces output, you refine your request based on what you see, and you iterate. This is not fire-and-forget. It’s working with the tool to produce what you actually need.

Notes on scope and applicability

Most agentic tools were built for software development, and they excel at tasks like writing code, debugging, building tools, and automating technical workflows. But the underlying capabilities—file manipulation, code execution, structured output, web search—are applicable to research tasks as well. You might use an agent to

  • Extract structured data from a collection of documents or images
  • Convert data between formats or restructure datasets
  • Generate interactive visualisations from raw data
  • Build searchable databases or interfaces for your research materials
  • Process and clean large collections of text or metadata
  • Create interactive teaching tools or demonstrations

The technology is general-purpose enough to be useful across disciplines, but most current interfaces assume programming knowledge that humanities scholars may lack. The field is working toward more accessible interfaces; what seems like a programmer’s tool today may be routine for non-technical users within a few years.

What can be built? — Examples

These projects were created using Claude Code or related agentic tools, with varying degrees of human involvement. They illustrate the range of what’s possible and the mix of agent and human labour involved. See the full portfolio for more examples and details.

CLA3131 QGIS Tutorials

Standard QGIS tutorials automatically redeveloped for ancient history case studies and written to RST format for embedding in the QGIS documentation interface. The agent handled the technical adaptation and formatting; the human provided the pedagogical brief, ancient history context, and reviewed the output for accuracy and disciplinary appropriateness.

The Lares

A comic short story collaboratively authored with an LLM (January 2025), imagining the consequences of a Classicist and a Theologian discovering an unexpected form of intelligence. This illustrates not a typical research application but the creative possibilities when human narrative direction meets agentic text generation.

Gnomon Shadow Explorer

Interactive demonstration of latitude calculation using a gnomon. Developed in 1–2 hours in January 2026. The agent wrote the code from a concept description; the human specified what to build, tested outputs, and iterated on functionality. Time investment was primarily in human direction and verification, not in manual coding.

Online RTI Viewer

A usable online viewer for Reflectance Transformation Imaging (RTI) files, allowing interactive exploration of artefact surfaces without specialised software. Demonstrates how agentic tools can make research materials more accessible to broader audiences.

Grammar Crammer

Experimental Latin grammar practice platform with diagnostic testing, targeted exercises, exit quiz, and progress tracking. A more substantial project illustrating how agents can build interactive educational tools with multiple interconnected features.

RavenC:LAW

Interactive map comparing place references across the Ravenna Cosmography and Peutinger Map. The underlying data and principal coding were done manually over three months. The codebase was subsequently improved in September 2025 with AI-assisted bugfixes and interface improvements. This illustrates how agentic tools can augment existing manual work rather than replacing it from scratch.

Interactive Ancient Globes

Ancient Greek and Islamic globes with adjustable orientation. Original coding took several weeks in summer 2024. Consolidated into a single app in September 2025 using agentic assistance, with celestial and solar rotation features added. Another example of AI augmenting prior manual work and enabling rapid feature development.

HestiaViz Redux

Revamped version of the HestiaVis tool for visualising place references in Herodotus’s Histories. Demonstrates interface redesign and feature improvement using agentic tools.

This guide itself

The LLM Guide for Humanities Scholars was developed using Claude Code, from initial planning through implementation. The human author provided the brief, editorial direction, and all factual and scholarly judgements. The agent handled page structure, HTML and CSS, drafting prose to specification, and iteration on design and content. The guide is a concrete example of agentic assistance applied to creating technical documentation and educational materials.

Risks and limitations specific to agentic AI

Chat-based LLM output is straightforward to evaluate: you read it and decide if it’s good. Agentic systems are harder to validate because the output isn’t a single text but a series of file modifications, computations, and decisions. This introduces new categories of risk.

Harder to verify than chat output

When an agent modifies twenty files, you can’t easily review each change the way you’d read and evaluate a chat response. Version control and diff tools become essential—not optional. You need to understand what changed, why, and whether the changes are correct. For large modifications, this verification work can be substantial. In some cases, you may find that verification would have taken less time than writing the agent instructions in the first place.

Scope creep

Agents can be overzealous. An agent told to “improve code quality” may refactor entire files you didn’t ask it to touch. An agent fixing a bug may add “helpful” features you didn’t request. An agent optimising performance may make changes that break something else. The autonomy that makes the tool powerful also enables it to make well-intentioned changes that produce unintended consequences. Clear constraints help, but they can’t eliminate the risk entirely.

Cost

Agentic workflows consume significantly more API tokens than simple chat, because the agent makes many internal calls, reads files repeatedly, and iterates. A straightforward chat query might cost pence. An agentic task that seems simple—“extract data from these documents and produce a summary”—might cost several pounds in API usage as the agent reads, processes, backtracks, and refines. Budget accordingly, particularly for large or iterative tasks. The financial barrier, while not prohibitive, is real.

Security and trust

Giving an AI agent access to your file system, terminal, or credentials requires trust in both the tool and the platform. You need to understand what permissions you’re granting. Can the agent read files outside a designated directory? Can it execute arbitrary commands? What happens to the data it processes? For work with sensitive materials—personal information, unpublished research, institutional data—these questions matter. If using cloud-based tools, your data passes through commercial servers. If using local tools, you maintain more privacy but sacrifice some convenience. There is no solution without tradeoffs.

The automation paradox

The more an agent does autonomously, the harder it is to maintain the critical oversight that makes the output trustworthy. There is a fundamental tension: speed and autonomy increase when you let the agent make decisions without constant oversight, but verification becomes harder. A task you give full autonomy to the agent might complete in minutes; a task you oversee step-by-step might take hours. But the step-by-step task may produce more reliable output because you’ve caught and corrected errors along the way. There is no clean resolution to this tension. You must choose based on the stakes: how critical is verification, and how much time can you invest?

Reproducibility

Agentic outputs may not be reproducible. Running the same task twice can produce different results because the agent’s decision-making process isn’t fully deterministic—even with the same input data and instructions, different intermediate steps might be taken. For research work that you’ll need to defend or reproduce, this is problematic. You can mitigate this by constraining the agent’s choices (more specific instructions) or by documenting exactly what happened (saving logs of the agent’s actions). But perfect reproducibility remains elusive.

The opacity problem

You can observe what an agent does—read the files it accesses, see the commands it runs, review the output it produces. But understanding why it made those specific choices is harder. The agent doesn’t have a fully transparent decision-making process you can inspect. It might make a sensible choice for inscrutable reasons, or a poor choice that seems reasonable until you see the results. This opacity isn’t unique to agentic systems—it’s inherent to how LLMs work—but it becomes more consequential when the agent is modifying your files and making decisions without explicit human approval at each step.

These limitations don’t make agentic tools useless. They make them tools that require care, attention to verification, and honest assessment of when they’re appropriate to use. For well-defined tasks where verification is manageable and stakes are modest, agentic tools can save significant time and effort. For open-ended tasks or high-stakes work, the oversight burden may outweigh the benefits.
