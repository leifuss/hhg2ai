## Task 1: Field Guides section

### Architecture

The core principle: **activities are the content, disciplines are the lens.**

AI tools don't differ by discipline — a classicist and a theologian use the same chat interface. What differs is which *tasks* matter, which *verification challenges* arise, and which *source types* are common. The Field Guides section is organised around tasks, with discipline tags that allow filtered browsing and short "pathway" pages that give each discipline a curated entry point.

### Tags configuration

Enable MkDocs Material's tags plugin. In `mkdocs.yml`:

```yaml
plugins:
  - tags:
      tags_file: field-guides/tags.md

# If not already present, ensure the tags plugin is in the plugins list.
# It ships with Material for MkDocs — no extra install needed.
```

Create `docs/field-guides/tags.md`:

```markdown
# Field Guides by Discipline

Browse activity guides filtered by discipline. Each activity page
is tagged with the disciplines for which it is most relevant.
```

This page auto-populates with a tag index generated by the plugin.

### Activity page template

Each activity page should follow a consistent structure. Here is the template — content will need to be written (see notes below on content sourcing).

```markdown
---
tags:
  - [relevant discipline tags]
---

# [Activity Name]

*March 2026*

## What this task involves

[2–3 paragraphs: what the scholarly task is, in terms a
humanities researcher would recognise. Not "how to use AI
for X" but "what X involves and where AI fits."]

## Where AI tools help

[Which capabilities matter: file upload, multimodal analysis,
long context, web search, code execution, etc. Reference
specific platform features where relevant, but keep it
cross-platform — link to Platform Guides for details.]

## What to watch out for

[Verification challenges specific to this task. Where do
models hallucinate? What kinds of errors are hard to catch?
What domain knowledge is needed to evaluate output?]

## Worked examples

[1–3 concrete examples. Each should show: what was asked,
what the model produced, and what the scholar then did
with it. Include at least one example where the output
needed significant correction.]

## Further reading

[Links to relevant Essentials pages (Verification, Prompting,
etc.), Platform Guides, and any external resources.]
```

### Pathway page template

Pathway pages are **short** — a paragraph of framing plus a curated, ordered list of activity guides. No duplicate content. They exist to answer: "I'm a classicist, where do I start?"

```markdown
# Classics & Ancient History

If you work in Classics or Ancient History, the AI-assisted
tasks you are most likely to encounter are translation and
language work, analysing images of inscriptions or objects,
working with fragmentary or lacunose texts, and spatial
analysis of the ancient world. Close reading and teaching
preparation are also relevant across the discipline.

## Suggested reading order

1. **Start here:** [Translation & Language](../translation.md)
   — covers ancient language work, checking model output
   against grammars and lexica, and the specific pitfalls
   of working with Greek, Latin, and other classical languages.

2. [Images, Objects & Visual Evidence](../images-objects.md)
   — working with photographs of inscriptions, pottery,
   coins, and archaeological material.

3. [Close Reading & Textual Analysis](../close-reading.md)
   — using AI for commentary, argument analysis, and
   engagement with secondary literature.

4. [Spatial & Geographic Analysis](../spatial.md)
   — mapping, gazetteers, and geographic reasoning about
   the ancient world.

5. [Teaching & Assessment](../teaching.md)
   — preparing seminars, tutorials, and assessments.

See also: [Quantitative & Tabular Data](../data.md) if you
work with catalogues, databases, or statistical evidence.
```

Each pathway page follows this pattern. Keep them to half a page. They should feel like advice from a colleague, not a syllabus.

### Content sourcing

The activity pages need substantive content. There are several sources:

1. **Existing site content.** The current Primer, Essentials, and Claude sections contain worked examples and task-specific advice that may be relocatable or adaptable. Check the existing "Worked Examples" pages in the Claude and ChatGPT sections — some of those workflows may map directly to activity pages.

2. **The existing "Field Guides" content.** The current site has field guides for Ancient Languages and Agentic AI under the Primer. The Ancient Languages content should be migrated to (or adapted for) the Translation & Language activity page. The Agentic AI content is platform-general and probably belongs in the Primer or Essentials rather than here.

3. **New content.** Most activity pages will need original writing. **For this initial implementation, create the file structure and templates with placeholder content.** Mark each activity page clearly as a stub:

    ```markdown
    !!! warning "Draft"
        This activity guide is under development. The structure
        is in place but the content is not yet complete.
    ```

    The Translation & Language page should be the most developed, since there is existing content to draw from. Others can be stubs.

- Update the Primer's nav to remove or redirect the old Field Guides sub-section, replacing it with a link to the new top-level Field Guides section.

## Task 2: Remove institution-specific content

### What to remove

Remove any Exeter-specific content from the public site. This likely includes:

- Pages or sections that reference University of Exeter policies, contacts, licences, or infrastructure
- Links to Exeter-specific training or support resources
- Any content that assumes the reader is at Exeter

### How to remove

1. **Identify Exeter-specific files.** Search the `docs/` directory for files containing "Exeter" (excluding the About page's author affiliation, which is fine to keep).

2. **For standalone Exeter pages:** Remove them from `mkdocs.yml` nav and delete or archive the files (move to a `_archive/` directory outside `docs/` if you want to preserve them for potential institutional reuse).

3. **For Exeter references within otherwise general pages:** Edit to remove the Exeter-specific content. If a section mixes general and Exeter-specific advice, keep the general and cut the local.

4. **The About page** (`docs/about/`) should retain Leif's affiliation ("University of Exeter") as author context. That's not institutional content — it's attribution.

### Future institutional content architecture

This is not for implementation now, but for reference when the question returns.

The cleanest approach for institutional customisation is a separate `mkdocs.yml` override per institution. The public site uses the base config. An institutional deployment uses:

```yaml
# mkdocs-exeter.yml
INHERIT: mkdocs.yml

nav:
  # ... includes everything from base nav, plus:
  - Your Institution:
    - institutional/exeter/index.md
    - Policies & Licences: institutional/exeter/policies.md
    - Local Support: institutional/exeter/support.md
    - Training: institutional/exeter/training.md
```

The institutional content lives in a separate directory (`docs/institutional/exeter/`) that can be:
- Maintained as a private add-on (charged or partnered)
- Included only in institutional builds, not the public site
- Templated so that adding a new institution means filling in a standard set of fields (institutional licence details, IT contact, local policies, etc.)

This architecture scales to multiple institutions without polluting the public site. But it requires MkDocs's `INHERIT` feature and some build pipeline setup, so defer until there's actual institutional demand.

## Nav position

In `mkdocs.yml`, "Field Guides" should sit after "Essentials" and before "Platform Guides":

```yaml
nav:
  - Home: index.md
  - The Primer: ...
  - Essentials: ...
  - Field Guides:
    - field-guides/index.md
    - Activities:
      - Translation & Language: field-guides/translation.md
      - Close Reading & Textual Analysis: field-guides/close-reading.md
      - Images, Objects & Visual Evidence: field-guides/images-objects.md
      - Archives & Manuscripts: field-guides/archives.md
      - Spatial & Geographic Analysis: field-guides/spatial.md
      - Quantitative & Tabular Data: field-guides/data.md
      - Teaching & Assessment: field-guides/teaching.md
    - Discipline Pathways:
      - Classics & Ancient History: field-guides/pathways/classics.md
      - Theology & Religious Studies: field-guides/pathways/theology.md
      - History: field-guides/pathways/history.md
      - Archaeology: field-guides/pathways/archaeology.md
      - Literature & Languages: field-guides/pathways/literature.md
      - Philosophy: field-guides/pathways/philosophy.md
  - Platform Guides: ...
  - Bootcamp: ...
  - Resources: ...
  - About: ...
  - Glossary: ...
```

**Rationale for position:** Field Guides come after Essentials because the Essentials (verification, prompting, data governance) are prerequisites. They come before Platform Guides because "what do I want to do?" is a more natural starting question than "which tool should I use?" For a reader working through the site in order, the sequence is: understand AI (Primer) → learn the ground rules (Essentials) → find your task (Field Guides) → choose your tool (Platform Guides) → go deeper (Bootcamp).

## Notes for the implementer

- The activity taxonomy (seven activities) is a starting point, not a final list. It should be easy to add new activity pages later without restructuring.
- Pathway pages should never contain content that doesn't also exist in an activity page. They are curated indexes, not content pages. If you find yourself writing substantive advice in a pathway page, it belongs in an activity page instead.
- The tags plugin generates the tags index automatically. You don't need to manually maintain the tags.md content beyond the heading — the plugin appends the tag listings.
- British English throughout.
- Activity pages that are stubs should be clearly marked with the draft warning admonition. Do not leave empty pages without explanation — a reader who navigates to a stub should understand that content is coming, not wonder if the link is broken.
- The `??? deeper` collapsible admonition (from the earlier technical content markers brief) should be used in activity pages where technical detail (e.g., specific API calls, code examples, tool configuration) appears within otherwise non-technical content.
