# A Hitchhiker's Guide to AI for the Humanities

## Site Architecture Brief — March 2026 (v2)

---

## 1. Identity

**Title:** A Hitchhiker's Guide to AI for the Humanities

**Repository:** `hhg2ai` (echoing the h2g2 convention from the Adams universe)

**Tagline candidates** (pick one or test):
- "Don't Panic."
- "Don't Panic. A practical guide to AI for humanities scholars."
- "Exploring the strangely useful universe of AI — with towel firmly in hand."

**Voice:** Warm, direct, occasionally wry. Rough Guide more than Lonely Planet — willing to name problems, not just attractions. The Adams reference sets the tone at the door but doesn't dominate every room. Most pages should read as calm, competent, human.

**Authorial transparency:** The guide is largely generated by Claude, and this should be stated clearly. The "Leif's Notes" admonition (see §6) provides a visible channel for the human author's editorial judgement, opinion, and caveats — distinguishing factual/procedural content from matters of interpretation and experience.

**What it is not:** An institutional guide. A corporate product tour. An AI advocacy site.

**What it is:** An independent, opinionated, practically useful resource for humanities academics navigating AI. Written by a working academic (with AI assistance, transparently acknowledged), maintained in the open.

## 2. Content Architecture

The site has four tiers, loosely mapped to the travel-guide model. Each tier has a distinct function and a distinct reading mode.

### Tier 1 — Before You Go (The Primer)

**Current location:** `/guide/`
**Purpose:** Conceptual orientation. What LLMs are, whether to care, how to think, what can go wrong.
**Audience:** Newcomers, sceptics, anyone who wants the big picture before touching a tool.
**Reading mode:** Discursive essay. Slow, reflective, no assumed knowledge.
**Travel analogy:** Country overview, history, culture, "is this trip for me?"

Pages (existing, largely stable):
- Briefing — what LLMs are and aren't
- Getting Started — first 15-minute exercise
- Mindset — how to think about these tools
- Use Cases — practical tasks by time investment
- FAQ — common anxieties and objections

**Field Guides** (discipline- and context-specific, living within the primer):
- Ancient Languages — translation, textual criticism, sacred texts
- Agentic AI — autonomous tools that plan and execute
- (Future: Medieval Studies, Theology, etc.)

Field Guides are contextual — they adapt the primer's general advice to specific disciplinary needs. They are lighter and more discursive than platform manuals. New fields can be added without touching the platform layer. This is also the layer where institution-specific adaptations would live, if that need arises.

### Tier 2 — Practical Information (Shared Essentials)

**Current location:** scattered across `/guide/rules.html` and `/claude/shared-principles.html`
**Proposed location:** `/essentials/`
**Purpose:** Cross-platform doctrine. Verification, data governance, citation, disclosure, prompting principles, multi-model strategy.
**Audience:** Everyone. Referenced from all platform manuals.
**Reading mode:** Reference. Scannable, procedural, with clear rules and checklists.
**Travel analogy:** Health & safety, money, language, customs, practical information.

Pages (consolidated from existing material):
- Verification & Citation — the verification ladder, citation practices, reference checking
- Data Governance — what to upload, what not to, institutional boundaries
- Prompting Principles — cross-platform prompting strategy
- Disclosure & Ethics — when and how to declare AI use
- Multi-Model Strategy — when to use which platform, comparative strengths
- Decision Sheet — one-page printable reference (existing)
- Cost & Plans — overview of pricing across platforms

**Why this tier matters:** This is the material that currently gets duplicated between the primer and the Claude guide, and would get duplicated again for ChatGPT and Gemini. Extracting it into a shared layer is the single most important architectural move. It makes the platform manuals leaner and the whole site more coherent.

### Tier 3 — Regional Guides (Platform Manuals)

**Current locations:** `/claude/` (exists), others planned
**Purpose:** Operational manual for a specific platform. How to use it, what each surface does, setup, limits, failure modes.
**Audience:** Someone who has decided (or been told) to use this platform and wants practical help.
**Reading mode:** Manual/reference. Task-oriented, procedural, matter-of-fact.
**Travel analogy:** Regional guide — "Southern Region," "Northern Region" — with local maps, recommended stops, practical tips.

Each manual follows a **common template** so readers can move between platforms by muscle memory:

```
[Platform] for Humanities Scholars

1. Overview & Which Tool Do I Need?
   - Decision flowchart
   - Minimum safe practice box
   - "You can safely ignore..." permissions
2. Conversational Interface
   - Setup, features, file handling, limitations
3. Projects / Persistent Context
   - What it is, when to use it, capacity
4. Agentic / Autonomous Features
   - Cowork, Canvas, Gems, etc. (platform-specific)
5. Coding / Building Tools
   - Code, Advanced Data Analysis, etc.
6. Integrations & Extensions
   - Browser, Office, third-party
7. Environment & Setup
   - Where to run it, prerequisites, plans & pricing
8. Worked Examples
   - Humanities-specific workflows
9. Glossary
   - Platform-specific terms
```

**Current manuals:**
- Claude (exists, needs refactoring to remove shared-principles material)

**Planned manuals:**
- ChatGPT (including GPT-4o, o-series, Canvas, Advanced Data Analysis)
- Gemini (including NotebookLM, AI Studio)
- Open-Weights Models (Llama, Mistral, Qwen — local deployment focus)
- Microsoft Copilot (short but full Tier 3 manual, expanded from existing primer page)

### Tier 4 — Itineraries (Training & Practice)

**Current location:** `/guide/bootcamp.html`
**Proposed location:** `/bootcamp/`
**Purpose:** Structured learning-by-doing. Guided exercises, increasing complexity.
**Audience:** Someone ready to invest focused time in building skills.
**Reading mode:** Tutorial. Step-by-step, with reflection prompts.
**Travel analogy:** Suggested itineraries — "3 days in Rome," "a week off the beaten track."

Pages:
- AI Bootcamp — ten structured experiments (existing)
- Quick Exercises — lighter tasks for workshops (potential)
- Teaching Scenarios — for those integrating AI into courses (potential)

The landing page replaces the current parking page. It should:

1. **Lead with "Don't Panic" and the brand identity.** A brief, warm, slightly wry welcome. One or two sentences establishing what this is and who it's for.

7. **Include a contribution invitation** — warm, travel-guide-inflected: "Fancy joining the expedition?" with a link to a conversation rather than a pull-request workflow.

8. **Not include:** Platform-specific branding, anything that makes this look like a single-vendor fan site.

## 5. Technical Stack

### Recommended: MkDocs + Material for MkDocs

**Why MkDocs Material:**
- Python-based (familiar, easy to maintain)
- Persistent left sidebar navigation (solves the "where am I?" problem)
- Built-in search across all pages
- Admonition/callout blocks (all six custom types — see §6)
- Content tabs (e.g., show Claude vs ChatGPT approaches side by side)
- Page status badges ("new", "updated", "needs review")
- Front matter support including `last_verified` dates
- Dark/light mode toggle
- Deploys to GitHub Pages via GitHub Actions
- Used by major documentation projects; mature, actively maintained
- Free and open source (MIT licence)

**What it gives you that raw HTML doesn't:**
- Sidebar navigation generated from directory structure
- Consistent layout without maintaining templates by hand
- Admonition blocks for the travel-guide "info boxes"
- Table of contents per page
- Full-text search
- Social cards (auto-generated preview images for sharing)
- Version/date stamps per page

**Customisation for the Hitchhiker's brand:**
- Custom colour palette (deep teal/petrol primary, amber accent — see §6)
- Custom logo and favicon
- Six custom admonition types including "Leif's Notes"
- Override the landing page template to be more magazine-style

### Migration path

1. Install MkDocs Material locally, create `mkdocs.yml` config
2. Move existing HTML content to Markdown files (mostly stripping HTML chrome, keeping content)
3. Set up nav structure in `mkdocs.yml`
4. Add custom CSS for brand colours and admonition types
5. Configure GitHub Actions to build and deploy on push
6. Test, redirect old URLs if needed
7. Go live

The content migration is the main work. The technical setup is an afternoon.

### Alternatives considered

| Option | Verdict |
|--------|---------|
| Keep raw HTML on GitHub Pages | Doesn't scale. No search, no nav, no callouts. |
| Jekyll (GitHub Pages default) | Worse developer experience for documentation. |
| Hugo | Go-based, overkill. Fewer documentation themes. |
| Docusaurus | React-based, heavier, aimed at software docs. |
| Ghost | Good for newsletters, wrong for a reference site. |


Easily changeable in MkDocs Material via `mkdocs.yml` or custom CSS overrides.

### Logo

**Current placeholder:** Teapot emoji (🫖) — a knowing Adams reference (Arthur Dent's tea obsession, the Nutrimatic machine, plus a faint echo of Russell's Teapot for epistemological resonance).

**Future:** Commission or generate a proper logo. Direction: something hand-drawn or slightly cartographic — a compass rose, a stylised map element, a teapot-as-compass, or an illustrated hitchhiker's thumb. Should feel independent and warm, not corporate. Nano Banana or similar image generation could produce candidates.

### Typography

- **Headings/UI:** Inter (clean, modern, highly legible)
- **Body text:** Source Serif 4 (warm, bookish, excellent readability)
- **Alternative:** System defaults (zero maintenance, perfectly fine)

### Custom admonition types

Six types, mapped to the travel-guide voice:

| Admonition | Icon | Colour | Use |
|-----------|------|--------|-----|
| `!!! panic "Don't Panic"` | 👍 | Teal | Reassurance. "This sounds complicated but you can ignore it." |
| `!!! essential "Essential"` | ✓ | Green | Must-know information. Minimum safe practice. |
| `!!! offtrack "Off the Beaten Track"` | 🗺️ | Amber | Advanced or adventurous options for the curious. |
| `!!! caution "Caution"` | ⚠️ | Red | Real risks, data governance warnings, things that can go wrong. |
| `!!! updated "Recently Updated"` | 🔄 | Blue-grey | Flags content that changed recently. |
| `!!! leifnote "Leif's Notes"` | ✏️ | Warm brown/sienna | Author's personal opinion, interpretive judgement, or navigational advice on thorny questions. Signals: "this isn't factual guidance — it's one academic's reading of the situation, offered to help you think it through." |

**"Leif's Notes" rationale:** Since the guide is largely AI-generated, there needs to be a visible, honest channel for the human author's editorial voice. Not every claim in the guide carries the same epistemic status. Some things are factual (Claude Pro costs $20/month). Some are procedural (upload your file, then ask a question). Some are matters of judgement (whether to use AI for a particular kind of scholarly work, how to weigh trade-offs, when "good enough" is good enough). The "Leif's Notes" admonition makes that distinction visible. It also models good practice — showing readers what it looks like to exercise independent judgement alongside AI assistance.

### The Map of the AI Landscape

**Concept:** A fictional/illustrated map of the AI domain as structured by the guide, in the spirit of the "Map of Mathematistan" (xkcd-adjacent style). The AI landscape rendered as geography: the Primer as a welcoming harbour or gateway city; the Essentials as a mountain range or river system running through everything; the platform manuals as distinct regions or islands (Claudeland, ChatGPTia, the Gemini Archipelago, the Open Weights Frontier); the Bootcamp as a trail or expedition route.

**Purpose:** Part wayfinding (helps readers see the whole site at a glance), part visual identity (distinctive, memorable, shareable), part tone-setter (signals that this guide doesn't take itself too seriously).

**Placement:** Landing page, prominently. Also usable as a social card image.

**Production:** Generate via Nano Banana or similar, then refine. Should be illustrative/cartographic rather than photorealistic. Hand-drawn or pen-and-ink aesthetic fits the brand better than polished digital art.

**Labelling:** Key landmarks should correspond to actual site sections so the map functions as visual navigation, not just decoration.

### Landing page design

The landing page breaks from the documentation layout. MkDocs Material supports custom page templates. Layout:
- Large "Don't Panic" header
- The Map (prominent, potentially interactive with clickable regions)
- Three pathway cards (primer, manuals, bootcamp)
- "What's new" or "recently updated" strip
- Mailing list signup
- Consultancy/workshops mention
- "Fancy joining the expedition?" contribution invitation

## 7. Staleness Detection Pipeline

### Factual checks (automated, monthly)

A GitHub Actions workflow that runs on a schedule:

1. Maintains a `claims.yml` manifest — a list of specific factual claims in the guides, each tagged with its source URL and the page it appears on. Example:
   ```yaml
   - claim: "Claude Pro costs $20/month"
     page: /claude/overview/
     source: https://www.anthropic.com/pricing
     last_verified: 2026-03-01
   - claim: "Projects available on all Claude plans"
     page: /claude/chat/
     source: https://docs.anthropic.com/en/docs/build-with-claude/projects
     last_verified: 2026-03-01
   ```
2. The action fetches each source URL, passes the claim + page content to an LLM, asks: "Is this claim still accurate according to the source? Reply YES, NO, or UNCLEAR."
3. Outputs a report: claims confirmed, claims that may be stale, claims that couldn't be verified.
4. Opens a GitHub Issue with the report if any claims are flagged.

**Cost:** Minimal. A few dozen LLM calls per month.

**What it catches:** Price changes, feature renames, plan restructuring, deprecated products, moved documentation.

**What it doesn't catch:** Conceptual staleness, shifted best practices, new products that should be covered.

### Conceptual review (manual, quarterly)

A calendar reminder. Spend one hour per quarter:
1. Scan each platform's changelog and blog for the past three months
2. Note anything that changes the guide's advice or emphasis
3. Update or flag pages as needed
4. Update `last_verified` frontmatter on reviewed pages

### Visible verification dates

Each page displays its `last_verified` date in the footer or header. Readers can see at a glance how fresh the content is. Honest, zero-cost, trust-building.

## 8. Analytics

**Recommended:** Plausible Analytics (privacy-friendly, no cookies, GDPR-compliant without consent banners, lightweight script). Paid service (~€9/month) but clean. Alternative: Umami (self-hostable, free).

**Fallback:** Google Analytics 4 (free, supported natively by MkDocs Material, but requires cookie consent and feeds the Google machine).

**What to track:**
- Page views by section/tier (which manuals get used?)
- Entry pages (where do people arrive?)
- Exit pages (where do they drop off?)
- Search queries (what are people looking for that they can't find?)
- Referral sources (where is traffic coming from?)

**Scheduled reporting:** A monthly summary emailed or generated automatically. Plausible supports email reports natively. If using GA4, a lightweight GitHub Action could pull the API and generate a summary.

**Purpose:** Prioritise maintenance effort, identify dead pages, understand audience, justify time investment.

## 9. Income Layer

### Immediate (zero setup)

- **"Buy me a coffee" link** in the site footer or about page. Ko-fi or GitHub Sponsors. Signals independence without creating obligations.

### Near-term (low setup)

- **Mailing list** via Buttondown (free up to 100 subscribers). Purpose: notify subscribers of major updates, new manuals, new exercises. A notification service, not an editorial newsletter.
- **Consultancy page.** "I run workshops on AI for humanities departments, libraries, and learned societies. Get in touch." Link to contact form or email. This is where real income potential lies.

### Future (only if demand appears)

- Occasional paid workshops (online or in-person)
- Institutional supporters acknowledged on the site
- Newsletter with paid tier (only if mailing list grows enough)

### What not to build

- A subscription product (premature, creates obligations)
- An ad-supported site (wrong incentives)
- A consultancy brand separate from the guide (the guide *is* the credibility)
- Bespoke tools or chat interfaces (scope creep)

## 12. Recommended Claude Code Skills

For the MkDocs build and ongoing maintenance, the following `/skills` would be most useful to install:

### High priority — install before starting

**`mkdocs-material`** — A skill for setting up, configuring, and customising MkDocs Material sites. Should cover: `mkdocs.yml` configuration, custom CSS overrides, admonition types, landing page template overrides, GitHub Actions deployment, plugin configuration (search, redirects, social cards, tags). This doesn't exist yet — worth creating via the skill-creator.

**`html-to-markdown`** — A skill for converting existing HTML pages to clean Markdown while preserving structure, links, tables, and semantic content. This is the core migration task. Again, worth creating — the conversion pattern is repetitive and well-suited to a skill.

### Already available and useful

**`frontend-design`** (`/mnt/skills/public/frontend-design/SKILL.md`) — Useful for the custom landing page template, any bespoke HTML/CSS overrides, and ensuring the visual design is polished.

**`skill-creator`** (`/mnt/skills/examples/skill-creator/SKILL.md`) — Use this to build the two custom skills above. The process: describe what the skill should do, draft it, test on a few sample pages, iterate.

**`canvas-design`** (`/mnt/skills/examples/canvas-design/SKILL.md`) — Potentially useful for generating the AI Landscape Map or logo candidates, though Nano Banana might be better for the cartographic style.

**`peer-review`** (`/mnt/skills/user/peer-review/SKILL.md`) — Already installed. Useful for getting multi-model feedback on draft content, especially the Essentials tier where cross-platform accuracy matters.

**`leif-work-context`** (`/mnt/skills/user/leif-work-context/SKILL.md`) — Already installed. Provides context for your DH work that helps Claude produce more relevant content.

### Nice to have later

**`staleness-checker`** — A custom skill for running the factual-check pipeline against `claims.yml`. Worth building once the site is live and the manifest exists.

**`content-auditor`** — A skill that scans Markdown files for common issues: broken internal links, missing frontmatter, stale `last_verified` dates, orphaned pages not in the nav. Useful for ongoing maintenance.
